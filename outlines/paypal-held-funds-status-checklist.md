# Page promise

- **Primary intent:** Diagnostic status triage for a PayPal notice that says funds are held, pending, unavailable, reserved, under review, or otherwise not withdrawable.
- **Reader outcome:** The reader records the exact status, registered account market, affected object, affected account functions, and the official authenticated route shown by PayPal—without guessing a cause, release date, document list, or outcome.
- **Proposed H1:** PayPal удержание средств: как разобрать статус и понять, что проверить первым
- **Promise boundary:** This is a status-interpretation checklist. It is not a 180-day release guide, ordinary post-sale payment-hold workflow, withdrawal troubleshooting guide, limitation appeal, verification-document guide, or promise of release.

# Adaptive-structure rationale

The query is diagnostic rather than time-specific. The reader starts with an ambiguous label, so the outline begins with exact wording and affected scope, then branches by mechanism before any market-specific example or action. A status classifier and an account-scope matrix carry the core value; a passed-period matrix is retained only as a conditional branch, never as a countdown.

The structure is intentionally different from `funds-held-180-days`: that page owns an explicit long-hold notice, a date/event check, and post-date balance states. This page owns first classification of an unfamiliar label. It also differs from `paypal-payment-hold-after-sale`: sale timing is only one possible context here and does not establish the mechanism. No universal causes, documents, timelines, release calculator, owner case, or standalone FAQ are included.

# Narrative order

1. Give a two-sentence qualified answer: the label alone does not identify a cause or deadline.
2. Capture the exact status text, date, named event, and account surface before interpretation.
3. Classify the affected object: one payment, balance/reserve, account, dispute/review, marketplace instruction, or withdrawal.
4. Compare sending, receiving, and withdrawing to determine whether the issue is transaction-level or account-wide.
5. Apply the registered-account market and account type before reading U.S., U.K., Ukraine/EU-facing, or other localized examples.
6. Follow only the official notice, balance, Resolution Center/Resolution Centre, or other authenticated route shown by the account.
7. If a displayed date has passed, compare the current state; do not calculate a universal release date.
8. Build one privacy-aware evidence snapshot and state the questions public sources cannot answer.
9. Offer narrow related-guide handoffs, then an optional individual review transition.
